# 🎮 DESIGN — направление и подход

Живой документ: куда движется игра, какие решения уже приняты, чего НЕ делаем.
Обновлять при каждом значимом решении по геймдизайну или архитектуре.

---

## 🎯 Что мы делаем

**Zelda-like, но не Zelda.** Top-down 2D, открытая карта, исследование, бой.

**Чего НЕ делаем:**
- ❌ Не RTS (StarCraft, Warcraft) — выделение групп, постройки, fog of war, build orders. Это другой движок, не «масштабирование того что есть».
- ❌ Не клон Zelda — сюжет, стиль, темп будут свои.
- ❌ Не повторяем `docs/scenario/` — те документы устарели как «космический шутер до Сталкера». Сюжет родится из играбельного прототипа, а не наоборот.

**Эстетика:** космос с нотками хоррора. **Это не жанр.** Эстетика ортогональна механикам — Dead Space (экшн), FTL (рогалик), Alien Isolation (стелс), Stellaris (стратегия) все «космос+хоррор», все разные. Сначала жанр механик, потом эстетика натянется.

**Жанр конкретно** — пока не фиксируем. Текущая база (twin-stick + враги + дроп + опыт + 4 оружия) подходит как минимум под:
- Arena shooter
- Top-down ARPG (Diablo-style)
- Survivor-like
- Zelda-like exploration

Жанр кристаллизуется через **прототипы**, не через выбор на бумаге. Vertical slice (см. BACKLOG) — место где это решится.

---

## 🧭 Принципы разработки

1. **Механики раньше сюжета.** Сначала играбельный билд, потом сценарий поверх него.
2. **Прототипы раньше архитектуры.** Не масштабируй то, чего ещё нет в игре.
3. **Data-driven раньше классов.** Когда контента много — данные в файлах, не классы для каждой единицы.
4. **Не делать преждевременную абстракцию.** Главный убийца pet-проектов.
5. **Эстетика — последний слой.** Сначала числа и поведение, потом спрайты и звук.

## 📐 Подход к масштабированию контента

### Где мы сейчас (v0.x)

| Контент | Кол-во | Паттерн |
|---|---|---|
| Оружие | 4 (меч / копьё / лук / бомба) | **Strategy** в `src/entities/weapons.py` |
| Враги | 3 (Light / Heavy / Fast) | Классы + Factory в `src/entities/enemy_factory.py` |
| Пикапы | 3 (Heart / Coin / XP) | Классы в `src/entities/pickup.py` |

**На текущем масштабе Strategy + Factory справляются.** Не трогаем.

### Когда переключаться на data-driven

Триггеры для рефакторинга:
- 🚦 Оружий стало **8+**, и они различаются параметрами, а не алгоритмом
- 🚦 Появились **модификаторы** («огненный меч», «лук точности»)
- 🚦 Появились **группы** в стиле Skyrim (light/heavy armor, iron/steel/daedric tier)

### Skyrim-style: один класс + теги

Skyrim не делает класс на каждый меч. Один `Item`, всё остальное — **данные** + **теги**:

```yaml
# weapons.yaml
iron_sword:
  base_class: MeleeWeapon
  damage: 1
  weight: 9
  value: 25
  tags: [sword, one_handed, iron, smithable]

steel_sword:
  base_class: MeleeWeapon
  damage: 2
  weight: 11
  tags: [sword, one_handed, steel, smithable]

daedric_sword:
  base_class: MeleeWeapon
  damage: 5
  tags: [sword, one_handed, daedric, smithable, magic]
```

Подсистемы (крафт, прокачка, инвентарь) реагируют **на теги**, не на типы:
- «+20% урон мечами» → фильтр `sword`
- «кузнец улучшает стальные» → фильтр `steel + smithable`
- «магия только на магических оружиях» → фильтр `magic`

Никаких новых классов.

### Diablo-style: affix-система поверх

Модификаторы как маленькие декораторы:

```python
sword.affixes = [
    Affix("flaming", damage_bonus=2, fire_dmg=1),
    Affix("of_swiftness", attack_speed=+0.2),
]
```

Каждый аффикс — записанный в данных модификатор. Система урона спрашивает у каждого: «дай свой вклад». Это и есть decorator pattern в живой природе.

### ECS — пока не трогаем

В `src/core/ecs/` остался скелет от удалённой multi-world системы. **Не использовать.** Подключим только если игра реально поедет в сторону «много типов сущностей с пересекающимся поведением» (юниты + постройки + мобы + NPC). Сейчас классы Player/Enemy/Pickup справляются.

## 🚫 Антипаттерны для этого проекта

- **Класс на каждую единицу контента** (`IronSword(Sword)`, `SteelSword(Sword)`, ...) — после 5–10 единиц превращается в ад. Сразу делать data-driven.
- **«Универсальный» Item-класс с 50 полями** — другая крайность. Базовый класс + теги + данные.
- **Глубокая иерархия наследования** (`Weapon → MeleeWeapon → Sword → MagicSword → FlamingMagicSword`). Заменять композицией / тегами / аффиксами.
- **GoF паттерны "ради паттернов"**. В геймдеве 90% классических паттернов в чистом виде не нужны. Работают: Strategy, Factory, Component composition, Observer (события). Остальные — по ситуации.
